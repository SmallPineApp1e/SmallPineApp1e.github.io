面试题03.数组中重复的数字

找出数组中重复的数字。

在一个长度为 n 的数组 nums 里的所有数字都在 0～n-1 的范围内。数组中某些数字是重复的，但不知道有几个数字重复了，也不知道每个数字重复了几次。请找出数组中任意一个重复的数字。

示例 1：

输入：
[2, 3, 1, 0, 2, 5, 3]
输出：2 或 3

限制：2 <= n <= 100000

思路：巧妙利用数组下标来匹配对应的元素，从前往后遍历数组，每个元素都要求下标与元素的值一一对应，如果在调换位置的过程中发现该位置的元素与别的位置的元素是相同的，那么就说明该数字重复，直接返回该数字

在可重复读隔离级别下, 启动一个事务就会产生一个read-view视图, 之后在事务提交之前, 都会使用这一个视图进行操作, 不会受其它事务操作表数据的影响

但是, 在上一篇文章: MySQL实战-行锁中提到, 如果两个事务A和事务B对同一行数据请求更新 那么后请求的事务B会进入阻塞状态, 等待前面的事务A释放行锁, 那么当事务B拿到行锁以后, 它取到的值到底是什么呢 ?是一致性视图的值还是被事务A更新后的值?

InnoDB的崛起之一：行锁

老规矩，学一样东西，就要知道为什么会有这项技术？为什么会出现行锁呢？不用行锁不行吗？

在没有行锁之前，访问表数据的时候使用的是表锁，其中有MDL锁和显式锁定(lock table)两种。每个请求线程访问一张表时（DML），都会将整个表锁住，之后的线程必须等待前面的线程释放表锁才可以去访问表，不仅如此，如果对表进行DDL操作，例如增删一个表的字段，也会将整张表锁住，可以看到并发度真的非常低。

所以，行锁应运而生，行锁就是针对数据表中行记录的锁。比如事务A需要更新一行数据，事务B也要更新同一行数据，事务B就必须等待事务A操作完成再进行更新。

行锁虽然有利于并发，但是对行锁的获取和释放时机的掌控是非常重要的，其中一个非常重要的协议是两阶段锁协议。

MySQL最重要的一个知识点——索引

MySQL为什么会出现索引这项技术？不用索引不行吗？

以一个真实的案例向你说明索引出现的原因

我们知道在没有索引的情况下，InnoDB执行引擎查找数据的方式是按行查找，从第一行开始，扫描每一行的id字段是否等于10，找到则返回该行数据；假设扫描一行需要0.1ms，在数据量很少的情况下，例如只有100个数据，上面这条语句只需10ms就执行完了；假设该表中含有100万行数据，并且age不是索引列，那么执行引擎就要按顺序扫描这100万行数据，如果运气好，第一行就找到了，但是绝大多数情况下不可能，所以有可能要扫描这100万行数据，每行需要0.1ms，最坏情况需要10万ms，这代价是巨大的。

所以索引的出现就是为了提高查询效率。像一本新华字典的目录一样，可以快速定位到数据所在位置并读取出来。

生动地给你讲解事务隔离的前世今生

事务就是要保证一组数据库操作，要么全部成功，要么全部失败。

在 MySQL 中，事务支持是在引擎层实现的。你现在知道，MySQL 是一个支持多引擎的系统，但并不是所有的引擎都支持事务。比如 MySQL 原生的 MyISAM 引擎就不支持事务，这也是 MyISAM 被 InnoDB 取代的重要原因之一。

当数据库上有多个事务同时执行的时候，就可能出现脏读（dirty read）、不可重复读（non-repeatable read）、幻读（phantom read）的问题，为了解决这些问题，就有了“隔离级别”的概念。
