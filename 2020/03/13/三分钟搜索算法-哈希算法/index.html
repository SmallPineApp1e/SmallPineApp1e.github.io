哈希算法

什么是哈希算法

将任意长度的二进制值串映射成一段固定长度的二进制值串，这种映射规则就是哈希算法。通过原始数据映射之后得到的二进制值串就是哈希值。

哈希算法满足的基本要求：

  • 从哈希值不能反向推导出原始数据（所以哈希算法也叫单向哈希算法）；

  • 对输入数据非常敏感，哪怕原始数据只修改了一个 Bit，最后得到的哈希值也大不相同；

  • 散列冲突的概率要很小，对于不同的原始数据，哈希值相同的概率非常小；

  • 哈希算法的执行效率要尽量高效，针对较长的文本，也能快速地计算出哈希值。

哈希算法的应用又非常多，例如：安全加密、数据校验、唯一标识、散列函数、负载均衡、数据分片、分布式存储。

1. 安全加密

最常用于加密的哈希算法是MD5（MD5 Message-Digest Algorithm，MD5 消息摘要算法）加密算法和SHA（Secure Hash Algorithm，安全散列算法）。

着重解释一下为什么散列冲突的概率要小而不是完全消除散列冲突，其实原理就是鸽巢原理，意思是只有10个鸽巢，但是有11只鸽子，所以有1个鸽巢必定会有2只鸽子；换而言之，哈希算法产生的哈希值的长度是一定的，比如以MD5加密算法为例，它生成的哈希值是128位的二进制串，所以能够表示的数据非常有限，最多能表示2^128个数据，而我们要加密的数据是无穷无尽的，所以当第(2^128)+1个数据进行MD5加密时（假设每个数据均不相同），必定会产生哈希碰撞。

下图是两串不同的字符串进行哈希算法哈希后发生哈希碰撞的哈希值

即便存在哈希碰撞的情况，但是概率非常的小，试想一想，用1个数据去进行MD5加密，哈希碰撞的概率就是1/2^128，所以MD5已经做的很好了。此外，强调一点的是，没有绝对的安全，只有更复杂、更安全的加密算法，破解时需要付出更大的代价。

2. 唯一标识

如果要在海量的图库中，搜索一张图是否存在，我们不能单纯地用图片的元信息（比如图片名称）来比对，因为有可能存在名称相同但图片内容不同，或者名称不同图片内容相同的情况。那我们该如何搜索呢？

粗糙的办法：我们知道图片本质上都是一串二进制码串，所以比较粗糙的办法就是拿这串二进制码串去服务器一张张图片对比二进制码串，如果对比成功就代表存在这张图片，但是如果一张图片非常的大，甚至几MB，此时效率会非常低下。

优雅的办法：可以给每张图片取一个信息摘要，例如取头部、中部和尾部各取50个字节，组合到一起进行哈希得到一个唯一的哈希码，然后将这个哈希码与文件的相关信息例如：文件路径、文件上传人等信息一同存储在一个哈希表当中，每次访问图片都可以先去哈希表里找有没有对应的图片标识（哈希码），如果不存在则认为该图片不存在，没有必要去请求服务器。

3. 数据校验

基于P2P协议下载文件时，会将一个文件分成很多不同的小块部分，例如1GB的文件，可以分成10个小块，每个小块100MB，从10台主机上下载文件，下载完以后再拼接起来变成一个文件。

问题是：如何保证每个小块的100MB文件没有丢失或者没有被宿主机恶意篡改呢？换而言之，要保证下载到本机的文件是安全、正确而且完整的。

在完整的文件中，每100MB采用一次哈希算法对这100MB的文件进行哈希，得到一个唯一的哈希码，保存在文件内部，由于哈希算法的敏感性，只要文件有一丁点改变，哈希码就会发生很大的变化，所以下载完文件之后，只要用相同的加密算法对下载的文件块进行哈希，如果对应部分得到的两个哈希码不相同则说明文件块被损坏了，需要从其它主机下载。

4. 散列函数

散列函数也是哈希算法的一种应用，它主要用于计算出元素放在哈希表中的哪个位置，散列函数对于哈希碰撞的要求并不是很高，它有许多解决办法，例如开放定址法与链表法，散列函数最关注于数据能否在哈希表中均匀分布，除此之外，散列函数执行的快慢，也会影响到哈希表的性能。

5. 负载均衡

那如何才能实现一个会话粘滞（session sticky）的负载均衡算法呢？也就是说，我们需要在同一个客户端上，在一次会话中的所有请求都路由到同一个服务器上。

我们可以通过哈希算法，对客户端 IP 地址或者会话 ID 计算哈希值，将取得的哈希值与服务器列表的大小进行取模运算，最终得到的值就是应该被路由到的服务器编号。 这样，我们就可以把同一个 IP 过来的所有请求，都路由到同一个后端服务器上。

6. 数据分片

1. 如何统计“搜索关键词”出现的次数？

假如我们有 1T 的日志文件，这里面记录了用户的搜索关键词，我们想要快速统计出每个关键词被搜索的次数，该怎么做呢？

两个难点，第一个是搜索日志很大，没办法放到一台机器的内存中。第二个难点是，如果只用一台机器来处理这么巨大的数据，处理时间会很长。

我们可以先对数据进行分片，然后采用多台机器处理的方法，来提高处理速度。

具体的思路是这样的：为了提高处理的速度，我们用 n 台机器并行处理。我们从搜索记录的日志文件中，依次读出每个搜索关键词，并且通过哈希函数计算哈希值，然后再跟 n 取模，最终得到的值，就是应该被分配到的机器编号。

2. 如何快速判断图片是否在图库中？

假如我们有 1T 的日志文件，这里面记录了用户的搜索关键词，我们想要快速统计出每个关键词被搜索的次数，该怎么做呢？

我们同样可以对数据进行分片，然后采用多机处理。我们准备 n 台机器，让每台机器只维护某一部分图片对应的散列表。我们每次从图库中读取一个图片，计算唯一标识，然后与机器个数 n 求余取模，得到的值就对应要分配的机器编号，然后将这个图片的唯一标识和图片路径发往对应的机器构建散列表。

7. 分布式存储

现在互联网面对的都是海量的数据、海量的用户。我们为了提高数据的读取、写入能力，一般都采用分布式的方式来存储数据，比如分布式缓存。我们有海量的数据需要缓存，所以一个缓存机器肯定是不够的。于是，我们就需要将数据分布在多台机器上。

该如何决定将哪个数据放到哪个机器上呢？我们可以借用前面数据分片的思想，即通过哈希算法对数据取哈希值，然后对机器个数取模，这个最终值就是应该存储的缓存机器编号。

如果数据增多，原来的 10 个机器已经无法承受了，我们就需要扩容了，比如扩到 11 个机器，这时候麻烦就来了。因为，这里并不是简单地加个机器就可以了。 原来的数据是通过与 10 来取模的。比如 13 这个数据，存储在编号为 3 这台机器上。但是新加了一台机器中，我们对数据按照 11 取模，原来 13 这个数据就被分配到 2 号这台机器上了。

因此，所有的数据都要重新计算哈希值，然后重新搬移到正确的机器上。这样就相当于，缓存中的数据一下子就都失效了。所有的数据请求都会缓存穿透，直接去请求数据库。这样就可能发生雪崩效应，压垮数据库。

要一种方法，使得在新加入一个机器后，并不需要做大量的数据搬移。这时候，一致性哈希算法就要登场了。
