多线程

线程与进程的区别与联系

进程：处于运行过程中的程序，并且有一定的独立功能。进程是资源分配和调度的独立单位。

进程的三个特征：独立性、动态性和并发性

独立性：进程拥有自己的地址空间，拥有自己独立的资源，在系统中是独立的实体。

动态性：进程与程序的区别在于，程序只是一个静态的指令集合，而进程则是正在系统中活动的一个指令集合。在进程中加入了时间的概念。

并发性：多个进程可以在单个处理器上并发执行，多个进程之间不会相互影响。

Q&A：并发性与并行性的区别？

并发性（concurrency）指在同一时刻只能有一条指令执行，但多个进程指令被快速轮换执行，在宏观上具有多个进程同时执行的效果。

并行性（parallel）指在同一时刻，有多条指令在多个处理器上同时执行。

线程（Thread）：也被称作为轻量级进程，线程是进程的组成部分，一个进程可以拥有多个线程，一个线程只能属于一个进程。线程拥有自己的局部变量、堆栈和程序计数器，它不拥有系统资源，它与父进程的其它线程共享该进程拥有的所有资源。多个线程存在于进程中可能会发生抢占CPU的情况。

归纳：一个操作系统可以执行多个任务，这些任务就是进程，每个进程可以拥有多个线程去共同完成任务。

线程的创建和启动

1. 继承Thread类创建线程类

2. 实现Runnable接口创建线程类

3. 使用Callable和Future创建线程

4. 创建线程的三种方式对比

实现Runnable接口与实现Callable接口的方式基本相同，只是Callable接口里的线程执行方法可以有返回值和抛出异常，因此可以将实现Callable和Runnable接口归为同一种方式。这种方式与继承Thread方式的主要差别如下。

采用实现Runnable、Callable接口方式创建多线程的优缺点：

优势：

线程类实现了Runnable或Callable接口还可以继承其它类

多个线程可以共享同一个target对象，适合多个相同线程来处理同一份资源

劣势：

编程较复杂，访问当前线程时需要用Thread.currentThread()方法。

采用继承Thread类的方式创建多线程的优缺点：

优势：

编写简单，访问当前线程时直接使用this即可。

缺点：

无法继承其它类。

因此一般推荐采用实现Runnable接口、Callable接口来创建多线程。

线程的生命周期

线程的五种状态：新建（New）、就绪（Ready）、运行（Running）、阻塞（Blocked）和死亡（Dead）。

1. 新建和就绪状态

新建状态：当程序使用new关键字创建一个线程后，线程处于新建状态。

就绪状态：程序调用线程的start()方法之后，该线程处于就绪状态。

Attention：启动线程使用start()方法而不是run()方法！永远不要调用线程对象的run()方法！调用start()方法，系统会把run()方法当作线程执行体来处理；如果直接调用run()方法，则run()方法被当作一个普通方法执行，系统会将线程对象当成一个普通对象，无法让多个线程并发执行。

2. 运行和阻塞状态

运行状态：处于就绪状态的线程获得了CPU，开始执行run()方法的线程执行体，则该线程处于运行状态。

阻塞状态：线程调用sleep()方法主动放弃占用的处理器资源、线程调用了一个阻塞式IO方法（键盘输入），在该方法返回前，该线程被阻塞、线程试图获得一个同步监视器，但是该同步监视器正被其他线程所持有、线程在等待某个通知（notify）、程序调用了线程的suspend()方法将该线程挂起。

阻塞 -> 就绪：sleep()方法的线程经过了指定时间、线程调用的阻塞式IO方法已经返回、线程成功获得同步监视器、线程正在等待某个线程时，其它线程发出了通知、处于挂起状态的线程调用了resume()恢复方法。

3. 线程死亡

线程会以三种方式结束（死亡）：run()或call()方法执行完成，线程正常结束、线程抛出一个未捕获的Exception或Error、直接调用该线程的stop()方法结束该线程-容易导致死锁。

测试线程是否死亡isAlive()：处于就绪、运行和阻塞状态时返回true，处于新建、死亡状态返回false。

线程死亡后不要尝试再”复活”线程，不要调用start()方法，会发生IllegalThreadStateExceptiion异常。

控制线程

1. join线程

join()方法的作用是让一个线程等待另外一个线程执行完成。当某个线程调用其它线程的join()方法时，该线程将会被阻塞，直到被调用join()方法加入的线程执行完毕。

重载方法：join(long millis)：等待被join的线程时间最长为milis毫秒。在millis毫秒后如果被join线程还没有执行完就不再等待。

2. 后台线程

后台线程又称为“守护线程”，典型的后台线程是JVM的垃圾回收线程。后台线程的设置必须在线程启动之前设置，即在start()前设置。

后台线程的特征：如果所有的前台特征都死亡，后台线程会自动死亡。

设置后台线程方法：setDaemon(true)

3. 线程睡眠：sleep

让当前执行的线程暂停一段时间并进入阻塞状态，可通过调用Thread类的静态方法sleep(long millis)来实现。在其睡眠时间内不会获得执行的机会，一直处于阻塞状态。

相似方法：yield()方法让当前正在执行的线程暂停，但是不会阻塞该线程，只是将该线程转入就绪状态。当线程调用yield()方法时，只有线程优先级大于等于当前线程优先级的其它线程才能获得执行机会。

sleep()与yield()的区别：

sleep()方法暂停当前线程后会给其它线程执行机会，不会理会线程的优先级问题；yield()方法只会给优先级相同，或优先级更高的线程执行机会。

sleep()方法由运行态转入阻塞态，直到经过睡眠时间后才会再次进入就绪态；yield()方法不会将线程转入阻塞态，只是强制将其转入就绪态，所有被yield()方法调用的线程完全有可能立即再次获得CPU资源被执行。

sleep()方法抛出了InterruptedException异常，而yield()方法没有声明抛出任何异常。

sleep()比yield()拥有更好的移植性，所以不推荐采用后者控制线程并发的执行。

4. 改变线程优先级

优先级高的线程获得较多的执行机会，而优先级低的线程则获得较少的执行机会。

每个线程的优先级与创建它的父进程的优先级相同，在默认情况下，main线程具有普通优先级。

Thread类提供setPriority(int newPriority)和getPriority()方法来设置和返回指定线程的优先级，Thread类提供三个常数：MAX_PRIORITY:10、MIN_PRIORITY:1、NORM_PRIORITY:5

3. 同步方法

同步方法就是使用synchronized关键字来修饰某个方法，同步方法的同步监视器是this，也就是调用该方法的对象。

不可变类总是线程安全的，因为它的状态不可改变。多线程环境下操作可变类就要额外的方法来保证线程安全。

不要对线程安全类的所有方法都进行同步，只对那些会改变竞争资源（共享资源）的方法进行同步。

如果可变类有两种运行环境：单线程环境和多线程环境

4. 释放同步监视器的锁定(释放锁)

两种特定情况下，线程不会释放同步监视器

程序调用Thread.sleep()、Thread.yield()方法来暂停当前线程的执行，当前线程不会释放同步监视器

在该线程执行同步代码块时，其它线程调用了该线程的suspend()方法将该线程挂起，该线程不会释放同步监视器（不会释放锁）。程序应该尽量避免使用suspend()、resume()方法来控制线程

5. 同步锁（Lock）

从Java 5开始，提供了一种功能更加强大的线程同步机制-Lock，它提供了比synchronized方法和synchronized代码块更广泛的锁定操作。

Lock是控制多个线程对共享资源进行访问的工具。线程开始访问共享资源之前应该先获得Lock对象。

Lock、ReadWriteLock是Java 5提供的两个根接口，并为Lock提供了ReentrantLock(可重入锁)实现类，为ReadWriteLock提供了ReentrantReadWriteLock实现类。

在实现线程安全的机制下，较常用的是ReentrantLock。通常使用try...catch..finally块来确保在必要时释放锁。获得可重入锁的线程可以对已被加锁的ReentrantLock锁再次加锁，ReentrantLock对象内部会维持一个计数器来追踪lock()方法的嵌套调用，线程每次调用lock()加锁后，必须显式调用unlock()来释放锁，所以一段被锁保护的代码可以调用另一个被相同锁保护的方法。

6. 死锁及常用处理策略

当两个线程相互等待对方释放同步监视器时就会发生死锁的情况，我们应该避免死锁发生，因为Java没有自动解决死锁的方法。

线程通信

1. 传统的线程通信

Object类提供的wait()、notify()和notifyAll()三个方法，由同步监视器对象来调用。

对于使用synchronized修饰的同步方法，因为该类的默认实例(this)就是同步监视器，所以可以在同步方法中直接调用这三个方法。

对于使用synchronized修饰的同步代码块，同步监视器是synchronized后括号里面的对象，所以必须使用这个对象调用这三个方法。

wait()：阻塞当前线程，直到其它线程调用本线程锁对象的notify()或者notifyAll()进行唤醒，使该进程进入就绪态。

notify()：唤醒在此同步监视器上等待的单个线程。如果由多个线程，则会随机唤醒其中一个线程，只有当前线程放弃同步监视器的锁（使用wait()方法），才可以执行被唤醒的线程。

notifyAll()：唤醒在此同步监视器上锁定的所有线程。只有当前线程放弃对该同步监视器的锁（使用wait()方法），才可以执行被唤醒的线程。

2. 使用Condition控制线程通信

Condition用于使用Lock对象来保证同步时的线程通信。Condition对象与Lock对象绑定在一起，要获得特定Lock实例的Condition实例，调用Lock的newCondition()方法即可。Condition提供如下三个方法：

await()：类似于wait()，导致当前线程阻塞，直到其它线程调用该Condition的signal()方法或者signalAll()方法。

signal()：类似于notify()，唤醒在此Lock对象上的某个线程，如果有多个线程在此Lock对象上等待，那么唤醒随机一个线程。只有当前线程放弃对该Lock对象的锁定之后（使用await()或者unlock()方法），才可以执行被唤醒的线程。

signalAll()：类似于notifyAll()，唤醒在此Lock对象等待的所有线程。只有当前线程放弃对该Lock对象的锁定之后（使用await()或者unlock()方法），才可以执行被唤醒的线程。

3. 使用阻塞队列（BlockingQueue）控制线程通信

Java 5提供了BlockingQueue接口，是Queue的子接口，它的主要作用是作为线程同步的工具，BlockingQueue具有以下特征：当生产者线程试图向BlockingQueue中放入元素时，如果该队列已满，则该线程被阻塞；当消费者线程试图从BlockingQueue中取出元素时，如果该队列已空，则该线程被阻塞。

BlockingQueue提供了两个支持阻塞的方法：

put(E e)：尝试把E元素放入BlockingQueue中，如果队列的元素已满，则阻塞该线程。

take()：尝试从BlockingQueue的头部取出元素，如果该队列的元素已空，则阻塞该线程。

BlockingQueue包含的方法之间的对应关系：

抛出异常 不同返回值 阻塞线程 指定超时时长
队尾插入元素 add(e) offer(e) put(e) offer(e, time, unit)
队头删除元素 remove() poll() take() poll(time, unit)
获取、不删除元素 element() peek() 无 无

常用的阻塞队列：

ArrayBlockingQueue：基于数组实现的BlockingQueue队列。

LinkedBlockingQueue：基于链表实现的BlockingQueue队列。

PriorityBlockingQueue：不是标准的阻塞队列，和PriorityBlockingQueue类似，它返回的是排好序的第一个元素，排序可以实现Comparable接口进行自然排序或者使用Comparator进行定制排序。

4. 线程池

线程池在系统启动时即创建大量空闲线程，程序将一个Runnable对象或者Callable对象传给线程池，线程池就会启动一个空闲线程来执行它们的run()或者call()方法，当方法执行结束后，该线程并不会死亡，而是再次返回线程池中成为空闲状态，等待执行下一个Runnable对象的run()或call()方法。

Java 5开始后，使用Executors工厂类来产生线程池，它包含以下方法：

newCachedThreadPool()：创建一个具有缓存功能的线程池，系统根据需要创建线程，这些线程将会被缓存在线程池中。

newFixedThreadPool(int nThreads)：创建一个可重用的、具有固定线程数的线程池。

newSingleThreadExecutor()：创建一个只有单线程的线程池，它相当于调用newFixedThreadPool()方法时传入参数为1

newSingleThreadScheduledExecutor()：创建只有一个线程的线程池，它可以在指定延迟后执行线程任务。

newScheduledThreadPool(int corePoolSize)：创建具有指定线程数的线程池，它可以在指定延迟后执行线程任务。corePoolSize指池中保存的线程数，即使线程是空闲的也被保存在线程池内。

ExecutorService newWorkStealingPool(int parallelism)：创建持有足够的线程的线程池来支持给定的并行级别，该方法还会使用多个队列来减少竞争。

ExecutorService newWorkStealingPool()：该方法是前一个方法的简化版本。如果当前机器有4个CPU，则目标并行级别被设置为4，也就是相当于为前一个方法传入4作为参数。

前三个方法返回一个ExecutorService对象，该对象代表一个线程池，它可以执行Runnable对象或Callable对象所代表的线程；

中间两个方法返回一个ScheduledExecutorService线程池，它是ExecutorService的子类，它可以在指定延迟后执行线程任务；

最后两个方法是Java8新增方法，这两个方法可充分利用多CPU并行的能力。生成的work stealing池，相当于后台线程池，如果所有的前台线程都死亡了，work stealing池中的线程会自动死亡。

ExecutorService代表尽快执行线程的线程池（只要线程池中有空闲线程，就立即执行线程任务），程序只要将一个Runnable

对象或者Callable对象（代表线程任务）提交给该线程池，该线程池就会尽快执行该任务。此对象提供了三个方法：

Future<?> submit(Runnable task)：将一个Runnable对象提交给指定的线程池，线程池将在有空闲线程时执行Runnable对象代表的任务，返回值Future可以调用isDone()/isCancelled()方法来获得Runnable对象的执行状态。

<T> Future<T> submit(Runnable task, T result)：第一个参数和上面的方法一样，第二个参数显式指定线程执行结束后的返回值，所以Future对象将在run()方法执行结束后返回result。

<T> Future<T> submit(Callable<T> task)：将一个Callable对象提交给指定的线程池，线程池将在有空闲线程时执行Callable对象代表的任务，返回值Future代表Callable对象里的call()方法的返回值。

ScheduledExecutorService代表可在指定延迟后或周期性地执行线程任务的线程池，它提供了下面四个方法：

ScheduledFuture<V> schedule(Callable<V> callable, long delay, TimeUnit unit)：指定Callable任务将在delay延迟后执行。

ScheduledFuture<?> schedule(Runnable command, long delay, TimeUnit unit)：指定command任务将在delay延迟后执行。

ScheduledFuture<?> scheduleAtFixedRate(Runnable command, long initialDelay, long period, TimeUnit unit)：指定command任务将在delay延迟后执行，而且以设定频率重复执行。也就是说，在initialDelay后开始执行，依次在initialDelay+period、initialDelay+2*period···处重复执行，依次类推。

ScheduledFuture<?> scheduleWithFixedDelay(Runnable command, long initialDelay, long delay, TimeUnit unit)：创建并执行一个在给定初始延迟initialDelay后首次启用的定期操作，随后在每次执行终止和下一次执行开始之间都存在给定的delay延迟。如果任务在任意一次执行遇到异常，就会取消后续执行；否则，只能通过程序来显式取消或终止该任务。

用完一个线程池后，应该调用该线程池的shutdown()方法，之后线程池不再接收新任务，但会将以前所有已经提交的任务执行完成。当线程池中的所有任务都执行完成后，池中的所有线程都会死亡；另外也可以调用线程池的shutdownNow()方法来关闭线程池，该方法试图停止所有正在执行的活动任务，暂停处理正在等待的任务，并返回正在执行的任务列表。

5. 使用线程池来执行任务的步骤

调用Executors类的静态工厂方法创建一个ExecutorService对象，该对象代表一个线程池；

创建Runnable实现类或Callable实现类的实例，作为线程执行任务；

调用ExecutorService对象的submit()方法来提交Runnable实例或Callable实例；

当不想提交任何任务时，调用ExecutorService对象的shutdown()方法来关闭线程池。

线程相关类

1. ThreadLocal类

ThreadLocal，是Thread Local Variable（线程局部变量）的意思，ThreadLocal的功用其实非常简单，就是为每一个使用该变量的线程提供一个变量值的副本，使每一个线程都可以独立地改变自己的副本，而不会和其他线程的副本冲突。

从线程的角度来看，就好像每一个线程都完全拥有该变量一样。

ThreadLocal提供了三个public方法：

T get()：返回此线程局部变量中当前线程副本中的值。

void remove()：删除此线程局部变量中当前线程的值。

void set(T value)：设置此线程局部变量中当前副本中的值。

ThreadLocal只是解决了多线程环境下的并发访问，因为它将不安全的变量拷贝多份分给不同的线程进行操作和访问，这样可以将不安全的变量转换为用ThreadLocal保证安全的变量。

但是ThreadLocal不能代替同步机制，两者面向的问题领域不同。同步机制时为了同步多个线程对相同资源的并发访问，是多个线程之间进行通信的有效方式；而ThreadLocal是为了隔离多个线程的数据共享，从根本上避免多个线程之间对共享资源的竞争，也就不需要对多个线程进行同步了。

通常建议：如果多个线程之间需要共享资源，以达到线程之间的通信功能，就使用同步机制；如果仅仅需要隔离多个线程之间的共享冲突，则可以使用ThreadLocal。

2. 包装线程不安全的集合

ArrayList、LinkedList、HashSet、TreeSet、HashMap、TreeMap等都是线程不安全的。可以使用Collections提供的类方法把这些集合包装成线程安全的集合。

<T> Collection<T> synchronizedCollection(Collection<T> c)：返回指定collection对应的线程安全的collection。

static <T> List<T> synchronizedList(List<T> list)：返回指定List对象对应的线程安全的List对象。

static <K, V> Map<K, V> synchronizedMap(Map<K, V> m)：返回指定Map对象对应的线程安全的Map对象。

static <T> Set<T> synchronizedSet(Set<T> s)：返回指定Set对象对应的线程安全的Set对象。

3. 线程安全的集合类

从Java 5开始，在JUC包下提供了大量支持高效并发访问的集合接口和实现类。可分为以下两类：

以Concurrent开头的集合类，如ConcurrentHashMap、COncurrentSkipListMap、ConcurrentSkipListSet、ConcurrentLinkedQueue和ConcurrentLinkedDeque。

以CopyOnWrite开头的集合类，如CopyOnWriteArrayList、CopyOnWriteArraySet。

其中以Concurrent开头的集合类代表了支持并发访问的集合，它们支持多个线程并发写入访问，这些写入线程的所有操作都是线程安全的，但读取操作不必锁定。以Concurrent开头的集合类采用了更复杂的算法来保证永远不会锁住整个集合，在并发写入时有较好的性能。

当多个线程共享访问一个公共集合时使用ConcurrentLinkedQueue是一个恰当的选择，它不允许存储null元素。

ConcurrentHashMap默认支持16个线程并发写入，当有超过16个线程并发向该Map写入数据时，有一些线程需要等待。实际上，程序通过设置concurrencyLevel构造参数可以支持更多的并发写入线程。因为ConcurrentHashMap和ConcerrentLinkedQueue支持并发访问，当使用迭代器对这两个集合迭代的过程中，如果有线程修改了元素，不会抛出异常（而HashMap和其它普通的集合则会抛出异常，因为fail-fast机制）。

由于CopyOnWriteArraySet的底层封装了CopyOnWriteArrayList，它的实现机制类似于CopyOnWriteArrayList集合。CopyOnWriteArrayList采用复制底层数组的方式实现写操作，当线程对其执行读取操作时，线程将会读取集合本身，无需加锁和阻塞。当线程对其进行写入操作时（包括add()/set()/remove()等方法），集合会在底层复制一份新的数组，接下来对新的数组执行写入操作，然后再赋值回原数组当中。由于CopyOnWriteArrayList集合执行写入操作时需要频繁地复制数组，性能比较差，但由于读操作和写操作不是操作同一个数组，而且读操作不需要加锁，因此读操作很快很安全。由此可见，CopyOnWriteArrayList集合适合用在读取操作远远大于写入操作地场景中，例如缓存等。
