索引的概念和实际用法

回到主键索引树搜索的过程，我们称为回表。可以看到，这个查询过程读了 k 索引树的 3 条记录（步骤 1、3 和 5），回表了两次（步骤 2 和 4）。

由于查询结果所需要的数据只在主键索引上有，所以不得不回表。那么，有没有可能经过索引优化，避免回表过程呢？那就是我们接下来提到的覆盖索引了。

由于覆盖索引可以减少树的搜索次数，显著提升查询性能，所以使用覆盖索引是一个常用的性能优化手段。

场景描述：在一个市民信息表上，是否有必要将身份证号和名字建立联合索引？

建立联合索引的目的在于减少回表的次数，假设现在有这样一个需要：根据身份证号查询出该公民的名字，那么建立联合索引就是有必要的，因为每个结点都包含唯一的身份证号和姓名作为key，不需要回表查找ID索引树的叶子结点。

如果为每个查询条件都设定一个索引对于磁盘空间来说是非常昂贵的，所以MySQL制定了一套索引的匹配规则，以力求减少索引树的数目。

最左前缀原则

B+ 树这种索引结构，可以利用索引的“最左前缀”，来定位记录。什么最左前缀，一脸懵逼啊？

简单来说，上图B+树结点的key排列顺序是（姓名，年龄），那么在索引匹配的过程中，必须按照从左到右开始匹配，如果只有年龄字段条件进行查询，则无法访问该索引树。

字符串的最左前缀规则

如果你要查的是所有名字第一个字是“张”的人，你的 SQL 语句的条件是"where name like ‘张%’"。这时，你也能够用上这个索引，查找到第一个符合条件的记录是 ID3，然后向后遍历，直到不满足条件为止。

只要满足最左前缀，就可以利用索引来加速检索。这个最左前缀可以是联合索引的最左 N 个字段，也可以是字符串索引的最左 M 个字符（’M%’）。

案例分析：在建立联合索引的时候，如何安排索引内的字段顺序？

评估的标准是：索引的复用能力。因为可以支持最左前缀，所以当已经有了 (a,b) 这个联合索引后，一般就不需要单独在 a 上建立索引了。

因此，第一原则是，如果通过调整顺序，可以少维护一个索引，那么这个顺序往往就是需要优先考虑采用的。

如果既有联合查询，又有基于 a、b 各自的查询呢？

查询条件里面只有 b 的语句，是无法使用 (a,b) 这个联合索引的，这时候你不得不维护另外一个索引，也就是说你需要同时维护 (a,b)、(b) 这两个索引。

这时候，我们要考虑的原则就是空间了。比如上面这个市民表的情况，name 字段是比 age 字段大的 ，那我就建议你创建一个（name,age) 的联合索引和一个 (age) 的单字段索引。

这个空间还是比较重要的！如果有100万行数据，假设name属性占8个字节，age属性占4个字节，如果反过来建立联合索引（age, name）并建立一个单字段索引（name），那么原来的两棵索引树耗费的空间至少是：(8 + 4 + 4) * 1024 * 1024 = 16MB，之后创建的两棵索引树所耗费的空间至少是：(8 + 4 + 8) * 1024 * 1024 = 20MB。

索引下推

MySQL 5.6 引入的索引下推优化（index condition pushdown)， 可以在索引遍历过程中，对索引中包含的字段先做判断，直接过滤掉不满足条件的记录，减少回表次数。

在(name, age)覆盖索引上直接过滤掉不满足name="张%"和age=“10”的数据行，只返回满足条件的数据行的ID进行回表。只需要回表2次。
