事务到底是隔离还是不隔离的

这篇文章是基于上一篇事务文章: MySQL实战-事务隔离)的MVCC这一个知识点进行展开说明事务到底是隔离的还是不隔离的

在可重复读隔离级别下, 启动一个事务就会产生一个read-view视图, 之后在事务提交之前, 都会使用这一个视图进行操作, 不会受其它事务操作表数据的影响

但是, 在上一篇文章: MySQL实战-行锁中提到, 如果两个事务A和事务B对同一行数据请求更新 那么后请求的事务B会进入阻塞状态, 等待前面的事务A释放行锁, 那么当事务B拿到行锁以后, 它取到的值到底是什么呢 ?是一致性视图的值还是被事务A更新后的值?

三个事务A B C按照上图的顺序启动事务并执行SQL语句

上面的第一条SQL语句表示立即启动事务, 注意与第二条启动事务的SQL语句作区分, 关键字consistent snapshot的意思是一致性快照, 意思是随着事务的启动并创建一个一致性快照, 这个快照就是一致性视图

使用start transaction只会在执行第一条快照读语句时才会执行, 也就是说并不是执行了这条启动事务语句, 就马上启动事务

先说结论: 事务 B 查到的 k 的值是 3，而事务 A 查到的 k 的值是 1

为什么是这样的结果呢? 事务B也有自己的一致性视图呀, 按道理来说更新k的值时原本k的值为1, 那么之后事务B读到的值是2才对

这篇文章就是解释在行锁和InnoDB事务之间穿插的一个MVCC的重要知识点

“快照”在MVCC下是如何工作的

在可重复读隔离级别下, 事务在启动时就拍了一个快照, 这个快照并不是把整库的数据都拷贝到一个快照当中, InnoDB里每个事务都有一个唯一的事务ID, 称为transaction id, 它在事务开始的时候向InnoDB引擎申请, transaction id是按申请顺序严格递增的

每一行数据都可能存在多个版本的, 每一个事务更新数据时, 都会生成一个新的数据版本, 然后把当前事务的版本号transaction_id赋值给当前这个数据版本的事务ID, 称为trx_id, 也就是要记录当前事务版本的数据是哪个事务操作生成的

同时, 旧的数据版本需要保留, 并且可以从新的数据版本中拿到旧的数据版本的信息

如上图所示, 就是同一行数据的某一列被多个事务更新后呈现出几个数据版本

从上图所示有4个数据版本, 当前的最新版本为V4, k的值为22, 是由transaction id = 25的事务更新的, 因此版本V4中, 这一行记录后面的trx_id列的值为25, 以此类推, 前面几个版本的行记录都会保存执行更新的事务transaction id

如何找到V1, V2, V3的数据呢 ?

当前的最新版本是V4, 所以理论上现在k = 22, 但是可重复读事务具有一致性的要求, 如果transaction id = 17的事务需要查找到k = 11, 就要通过undo log(回滚日志)了, undo log就是上图的U3 U2 U1, 如果事务id = 17需要再次查找k的值, 则会根据当前版本和undo log计算出k的值; 同理, 如果需要V2版本的k值, 就通过V4依次执行U3 U2来计算出来

所以, 如果一个事务需要查找当前事务的值, 就会去从当前最新的数据版本逆向遍历这条链, 如果找到transaction id = 自己则返回当前版本的值, 如果找到某个数据版本的事务id1大于自己的事务id2, 则说明这个事务是在自己之后创建的, 不能接受这个版本的值, 因为可重复读是在我创建事务后保证其它事务的更新对自己是不可见的, 如果找到某个数据版本的事务id1小于自己的事务id2, 那么认为这个事务是在自己前面创建的, 这个数据版本的值可以接受, 就读取当前数据版本id1的值

在MySQL中如何维护这个数据版本链呢?

InnoDB为每个事务都构造了一个数组, 用于保存当前这个事务启动的那一刻, 当前数据库正在“活跃”的事务ID, “活跃”指的是启动了但尚未提交的事务

如上图所示, 数组里面事务ID的最小值记为低水位, 当前系统已创建事务ID的最大值 + 1为高水位, 这个视图数组和高水位组成了当前事务的一致性视图(read-view)

而数据版本的可见性规则, 就是基于数据的row_trx_id和这个一致性视图的对比结果得到的

一个数据版本的row trx_id, 可能有以下几种情况 :

如果trx_id落在绿色部分, 表示这个版本是已经提交的事务或者当前这个事务自己生成的, 这个数据是可见的;

如果落在红色部分, 则表明这个版本是由将来启动的事务生成的, 肯定是不可见的

如果落在黄色部分, 则分为两种情况:

  1. 如果row trx_id存在于数组里, 那么这个版本是由还没有提交的事务生成的, 不可见;
  2. 如果row trx_id不在数组中, 表示这个版本已经提交了的事务生成的, 可见;

这样听可能会很模糊, 或许举个例子会更加清楚 :

对于上图来说, 如果有一个事务D, 它的低水位为18, 那么当它访问这一行数据时, 就会先判断V4的trx_id , 因为trx_id = 25的数据版本比低水位要大, 则认为该数据版本是还没提交的事务生成的, 数据是不可见的, V4通过U3计算出V3, 比较V3的trx_id = 17, 小于当前的低水位所以在它看来, 这一行的值是11,

观察下图, 分析事务A的返回结果, 我们作以下假设:

事务A开始前, 系统里只有一个活跃事务ID是99

事务A B C的版本号分别是100, 101, 102, 当前系统只有这4个事务

三个事务开始之前, 这一行数据的row trx_id为90

事务 A 的视图数组就是[99,100], 事务 B 的视图数组是[99,100,101], 事务 C 的视图数组是[99,100,101,102]。

第一个有效更新是事务 C，把数据从 (1,1) 改成了 (1,2)。这时候，这个数据的最新版本的 rowtrx_id 是 102，而 90 这个版本已经成为了历史版本。

第二个有效更新是事务 B，把数据从 (1,2) 改成了 (1,3)[这里有点怪~]。这时候，这个数据的最新版本（即 row trx_id）是 101，而 102 又成为了历史版本。

事务 A 查询语句的读数据流程是这样的:

找到(1, 3)时, 比较row trx_id = 101大于高水位100, 所以认为该数据是在之后生成的, 数据不可见

比较(1, 2)的row trx_id = 102大于高水位100, 同样不可见

比较(1, 1)的row trx_id = 90, 小于低水位99, 认为是之前已经提交的数据, 数据可见, 所以事务A读取到的值是(1, 3)

这样执行下来，虽然期间这一行数据被修改过，但是事务 A 不论在什么时候查询，看到这行数据的结果都是一致的，所以我们称之为一致性读。

对于一个事务视图来说，除了自己的更新总是可见以外，有三种情况：

版本未提交，不可见；

版本已提交，但是是在视图创建后提交的，不可见；

版本已提交，而且是在视图创建前提交的，可见。

补充说明:

在真实的MySQL数据库中, 低水位的名称是up_limit_id, 高水位是low_limit_id, 当前的事务id字段为creator_trx_id

维护当前活跃的事务id数组称为trx_ids

如果当前事务查询行记录:

如果数据版本大于up_limit_id则数据不可见 , 因为是在本事务之后创建的;

如果小于low_limit_id就认为该事务已经提交了, 数据是可见的;

如果处于[low_limit_id, up_limit_id]之间:

  • 如果在当前的一致性视图里找不到这个事务id, 就说明该事务已经提交了, 数据可见
  • 如果在当前的一致性视图中能找到这个事务id, 那么说明该事务还未提交, 数据不可见

更新逻辑

事务 B 的 update 语句，如果按照一致性读，好像结果不对？

如果事务B在B作更新前获取值, 是不会看到(1, 2)这个值的

事务 B 的视图数组是先生成的，之后事务 C 才提交，事务B应该无法看见事务C做的更新操作, 但事实上事务B看见了, 这就是更新操作的当前读逻辑

当前读: 更新数据都是先读后写的, 而这个读, 只能读当前最新版本的值, 称为”当前读”

所以, 在更新的时候，当前读拿到的数据是 (1,2)，更新后生成了新版本的数据 (1,3)，这个新版本的 row trx_id 是 101。

如果事务C不是马上提交的, 变成事务C’, 那么事务B进行更新操作会出现什么现象呢?

事务C’不是马上提交事务, 这时候就要联想到事务的”两阶段锁协议”了, 事务在DML操作时会给行记录加锁

读锁之间不会互斥, 因为对同一行记录可以多个线程去请求

读写锁之间 写锁之间互斥, 也就是保证每一次只有一个线程对行记录进行更新操作

事务 C’没提交，也就是说 (1,2) 这个版本上的写锁还没释放。而事务 B 是当前读，必须要读最新版本，而且必须加锁，因此就被锁住了，必须等到事务 C’释放这个锁，才能继续它的当前读。

到这里，我们把一致性读、当前读和行锁就串起来了。

回到文章开头的问题: 事务的可重复读的能力是如何实现的

可重复读的核心在于一致性读(consistent read); 而事务对行记录进行更新时, 只能使用当前读, 当前记录的行锁被其它事务占用, 那么只能阻塞进行锁等待

读提交隔离级别下的一致性视图

读提交隔离级别下, 每条语句在执行时都会生成一个一致性视图

下面来分析分析在度提交隔离级别下, 各个事务的查询语句查到的k是什么呢?(有趣~)

事务A B C依次启动, 数据库里存在一个版本号为90的行记录

事务C执行更新操作并提交事务, 版本号为102, 行记录为(1, 2)

事务B此时更新k值为3, 版本号为101, 行记录为(1, 3), 然后执行查询语句时生成另一个一致性视图, 一致性视图为[100, 101], 高水位为103, 查询到第一个的行记录版本号为自己的事务id, 所以事务B读到的k为3

然后事务A执行查询语句, 在执行时也会生成一个一致性视图, 一致性视图数组为[100, 101], 此时由于事务B还没有提交事务, 所以(1, 3)是不可见的, 而事务C的版本号是102, 小于高水位103, 且不在活跃数组中, 所以事务C已经提交事务, 此时认为版本号102的数据(1, 2)是可见的, 所以事务A查询的k值为2

小结

InnoDB 的行数据有多个版本，每个数据版本有自己的 row trx_id，每个事务或者语句有自己的一致性视图。普通查询语句是一致性读，一致性读会根据 rowtrx_id和一致性视图确定数据版本的可见性。

对于可重复读，查询只承认在事务启动前就已经提交完成的数据；

对于读提交，查询只承认在语句启动前就已经提交完成的数据；

当前读，总是读取已经提交完成的最新版本。更新语句就是利用当前读来获取最新版本的数据
