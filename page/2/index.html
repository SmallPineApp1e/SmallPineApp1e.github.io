理解Cookie和Session

为什么会出现Cookie和Session

As we all know， HTTP是一种无状态的协议，那么服务器是如何分辨出请求是谁发起的呢？或者我们这样想，服务器怎么识别到不同的用户发起请求呢？所以为了解决这个问题，Session和Cookie应运而生。

Session和Cookie典型应用场景

登录操作：输入用户名和密码登录后，很多情况下在一段时间内，关闭浏览器再重新打开就不需要再次手动输入用户名和密码登陆了，这个时候用到的一个机制就是Cookie和Session，使用Session在服务器保存状态信息，使用Cookie在客户端保存sessionId，当用户再次访问网站时，会自动将Cookie当中的sessionId传递给服务器端，服务器端识别到这个id是有效的，那么会返回用户信息给客户端进行保存。

Cookie和Session的作用

Cookie

Cookie是将所有的数据均保存在客户端，比如，当你登录以后，服务器设置了cookie为，那么当你再次请求服务器时，浏览器会将用户名一起发给服务器，这些变量有一定的特殊标记，服务器会识别为cookie。只要不关闭浏览器，cookie变量就是有效的，所以能够保证长时间不掉线。如果能够截取到某个用户的cookie变量，然后伪造一个数据包发送到服务器，那么服务器依然会认为是合法的请求。所以Cookie被攻击的可能性比较大。如果设置了有效时间，那么它会保存在客户端的硬盘上，下次再次访问该网站时，浏览器会先检查有没有Cookie，如果有则读取该Cookie，然后发给服务器。

Get和Post方法

👨‍⚖️面试官：小伙子听说你计算机网络整挺好，你知道Get和Post的区别吗？

👦我：

  1. Get在浏览器回退时是无害的，但是Post回退时会再次提交请求。
  2. Get请求会被浏览器主动cache，而Post不会，除非主动设置。
  3. Get请求只能进行URL编码，而Post支持多种编码方式。
  4. Get请求参数会被完整保留在浏览器历史记录里，而Post请求中的参数不会被保留。
  5. Get参数通过URL传递，而Post通过RequestBody传递
  6. Get请求在URL中传送的参数是有长度限制的，而Post没有

进程的通信方式

为什么需要进程通信

每个用户进程都拥有各自不同的用户地址空间，任何一个进程的全局变量在另外一个进程中都无法看到，所以进程间通信必须要通过内核，在内核中开辟一块缓冲区，进程A把数据从用户空间拷贝到内存缓冲区，进程B再从内核缓冲区把数据读走，内核提供的这种机制叫做进程间通信。

进程间通信的本质

进程之间可以看到一份公共资源，提供这份资源的形式或者提供者不同，造成了通信方式不同，而pipe(管道)就是提供这份公共资源的形式之一。

管道通信

匿名管道

创建方式：调用pipe(int fd[2])函数创建管道，fd参数返回两个文件描述符，fd[0]指向管道的读端，fd[1]指向管道的写端，fd[1]的输入等于fd[0]的输出。

KMP算法入坑到脱坑

KMP 算法（Knuth-Morris-Pratt 算法）是一个著名的字符串匹配算法，效率很高，但是确实有点复杂。

在开始学习KMP算法之前约定，用pat表示模式串，长度为M，用txt表示主文本串，长度为N。

KMP算法时在txt中查找字串pat，如果存在，则返回这个字串的起始索引，否则返回-1

本文的骚操作：KMP算法就是一个动态规划问题。对于动态规划，最重要的是确定状态，也就是dp数组，是一维（一个状态）还是二维（两个状态）······

我见过的 KMP 算法应该是，一波诡异的操作处理 pat 后形成一个一维的数组 next，然后根据这个数组经过又一波复杂操作去匹配 txt。时间复杂度 O(N)，空间复杂度 O(M)。其实它这个 next 数组就相当于 dp 数组，其中元素的含义跟 pat 的前缀和后缀有关，判定规则比较复杂，不好理解。 （是我实锤了）

本文则用一个二维的 dp 数组（但空间复杂度还是 O(M)），重新定义其中元素的含义，使得代码长度大大减少，可解释性大大提高。 本文的代码参考《算法4》（md有空一定要去看完这本书）

1. HTTP和HTTPS的发展历史

什么是HTTP协议？

HTTP是一种超文本传输协议，一种基于请求和响应、无状态的、应用层的协议，常用于TCP/IP协议传输数据，是互联网上应用最为广泛的一种网络协议，所有的WWW文件都必须遵守这个标准。设计HTTP协议的初衷是为了发布和接收HTML页面的方法。

HTTP协议的发展历史

版本 产生时间 内容 发展现状
HTTP/0.9 1991年 不涉及数据包传输，规定客户端与服务器之间的通信格式，只能使用GET请求 没有作为正式的标准
HTTP/1.0 1996年 传输内容的格式不受限制，增加了PUT、PATCH、HEAD、OPTIONS、DELETE命令 正式作为标准
HTTP/1.1 1997年 增加了持久连接（长连接）、节约带宽、Host域、管道机制、分块传输编码 2015年使用最为广泛
HTTP/2.0 2015年 多路复用、服务器推送、头信息压缩、二进制协议等 逐渐覆盖市场

三分钟操作系统-进程、线程与协程的区别与联系

发表于 2020-03-24 | 分类于 操作系统
字数统计 904字 | 阅读时长 3分钟

进程

进程是动态的，而程序是静态的，进程就是将保存在硬盘上的程序运行在内存当中，进程在内存空间当中会存在一个独立的内存体，该内存体有自己的地址空间、有自己的堆，上级挂靠的单位是操作系统。操作系统是以进程为单位分配系统资源的（CPU时间片、内存等资源），进程是资源分配的基本单位。

​

线程

线程，有时候也会被称为轻量级进程（Lightweight Process，LWP），是CPU调度执行的最小单位。

​

进程与线程的区别与联系

联系：

线程属于进程，一个线程只能属于一个进程，一个进程可以拥有多个线程。

系统资源分配给进程，同一个进程的所有线程共享该进程的所有资源。

线程的运行是由CPU进行分配的，真正运行在处理器上的是线程。

线程在执行过程中，需要协作同步。不同进程的线程间要利用消息通信的办法实现同步。

区别：

调度：线程是CPU调度和分配的基本单位，进程是操作系统分配资源的基本单位。

并发性：不仅进程之间可以并发执行，同一个进程的多个线程之间也可并发执行；

拥有资源：进程是拥有资源的一个独立单位，线程不拥有系统资源

系统开销：在创建和撤销进程时，系统都要为其分配和回收资源，导致系统的开销明显大于创建或撤销线程时的开销。进程有独立的地址空间，一个进程崩溃以后不会影响到其它进程。而线程没有自己的独立地址空间，有自己的堆栈和局部变量，一个进程挂掉时这个进程对应的所有线程都会挂掉。

白话理解进程与线程

计算机的核心就是CPU，它承担着所有的计算任务，它就像一座工厂，时刻在运行。

​

工厂的电力有限，每次只能够运行一个车间，也就是说如果有某一个车间在运行时，其它车间必须停止运作，也就是说在单核CPU中，同一时刻只能有一个进程在运行。

​ 进程就好比工厂的车间，它代表CPU所能处理的单个任务。任一时刻，CPU总是运行一个进程，其他进程处于非运行状态 。

​

一个车间里，可以有很多工人。他们协同完成一个任务。线程就好比车间里的工人。一个进程可以包括多个线程。

车间的空间是工人们共享的，比如许多房间是每个工人都可以进出的。这象征一个进程的内存空间是共享的，每个线程都可以使用这些共享内存。

​

协程

协程可以是“主动式”的，由程序来告诉计算机，我要进入IO行为了，可以把CPU的控制权交给别人，这个实现的基础是用户态的。所以，协程在用户线程中，两个程序协商好了，通过某种方式协作运营，共享CPU控制权的方法。一般来说，这种协商的方法通用的关键字是yield。

OSI七层模型及各层的作用

OSI的七层模型：

应用层、表示层、会话层、运输层、网络层、数据链路层、物理层

物理层：底层的物理硬件如电缆、双绞线······传输的信号是电信号，根据约定规定电信号的大小转化为二进制位0和1。

数据链路层：对传输过来的电信号进行封装成帧，在两个相邻的结点上透明地传输帧数据，如果检验到帧出现错误则丢弃该帧，无法进行校正。

网络层：使用无连接的网际协议IP和许多路由选择协议，为分组交换网上的不同主机提供通信服务，把运输层产生的报文段或用户数据报封装成分组或包进行发送。

运输层：向两个主机中的进程提供端到端的通信服务，最广泛使用的协议是TCP协议和UDP协议，TCP协议是可靠传输，数据包的长度没有限制，但为了保证网络的效率，TCP数据包的长度不会超过IP数据包的长度，以确保单个TCP数据包不会再被分割。UDP协议的报头是8个字节，它是不可靠传输，从长度不能超过65535个字节，也就是一个IP数据包的长度。

应用层： 直接为用户的应用进程提供服务，规定了不同的应用进程的数据格式，如 HTTP、支持文件传输的 FTP 协议等。
